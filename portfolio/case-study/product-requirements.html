1. Executive summary

TalentHub is an applicant tracking system delivered as an npm monorepo with four runtime applications supporting the full core hiring loop: publish jobs → discover and apply → triage in a Kanban pipeline → progress through interview and offer stages → hire or reject.

Candidate portal Port 3000

Public job seekers browsing open roles.

My Applications Port 3002

Registered candidates managing profile, CV, and applications.

Backoffice Port 3001

Internal recruiters and hiring staff.

Central API Port 4000

Shared authentication and candidate account services.

Job data, applications, and staff workflows persist in PostgreSQL via Prisma. Most domain logic runs in Next.js route handlers (BFF pattern); the central Hono API handles auth and candidate registration only. This PRD distinguishes implemented, partial, and planned capabilities.

2. Vision and objectives

2.1 Vision

Give hiring teams a focused backoffice to triage and progress applications efficiently, while giving candidates a clear path from job discovery to application tracking — without fragmenting context across jobs, candidates, and pipeline views.

2.2 Product objectives

  1. Reduce time-to-triage — Staff review who applied, for which job, when, with what documents, from a single applications hub.
  2. Enforce valid hiring workflow — Status changes follow a transition matrix with audit logging and controlled reopen flows.
  3. Enable self-service candidate onboarding — Register, verify email, upload CV, apply without staff intervention.
  4. Maintain a single source of truth — One PostgreSQL schema shared across all apps; shared validators and types.
  5. Support local-first development — File uploads to disk, optional OpenAI for CV parsing and relevance scoring.

2.3 Design principles

  • Progressive disclosure — Summary in list/pipeline; depth on application and candidate detail pages.
  • Application as unit of work — Candidate and job are linked contexts, not separate workflows.
  • Honest system state — Disable or block impossible transitions rather than failing silently.
  • Implementation-aligned documentation — Specs track the Next.js apps, not legacy static markup alone.

6. Application lifecycle

6.1 Status enum (10 values)

* Rejected allows controlled reopen to under_review or shortlisted only.

6.2 Active Kanban columns

Submitted → Under Review → Shortlisted → Interview Scheduled → Interview Completed → Offered → Hired

Rejected and Withdrawn appear as separate tabs or filtered views.

6.3 Business rules (summary)

  • Reject requires a reason.
  • Withdraw requires a withdrawal source.
  • Hired from Offered requires offerAccepted: true.
  • Interview Scheduled requires an application_interviews row.
  • Transitions from Rejected use reopen endpoint, not standard PATCH.
  • Hired and Withdrawn are terminal for normal PATCH.
  • Optimistic concurrency via optional expectedUpdatedAt (409 on conflict).

8. Non-functional requirements

8.1 Accessibility

Target WCAG 2.2 Level AA for Next.js apps. See WCAG 2.2 audit. Known gaps: modal focus traps (TH-191), placeholder legal footer links (TH-192), pipeline live-region announcements.

8.2 Privacy and data protection

PDPA/GDPR-oriented audit: GDPR / PDPA audit. Candidate PII in profile, CV files, and application attachments. Staff sessions via httpOnly cookies; candidate tokens in localStorage (TH-193 HttpOnly migration).

8.3 Security

  • Password hashing (bcrypt) for staff and candidates.
  • Account lockout on failed login attempts (API).
  • Staff routes gated by middleware; candidate routes require Bearer JWT.
  • File upload type and size validation on CV endpoints.

8.4 Performance and scale

Local/dev-oriented: filesystem storage, single PostgreSQL instance. Pipeline loads up to 500 applications server-side. No CDN, email queue, or horizontal scaling documented yet.

8.5 Design system

HTML catalog: design-system/index.html. Tokens, components, and layout templates aligned with apps/*/src/styles.

9. User journeys (as-built)

9.1 Candidate: discover and apply

Browse jobs (portal :3000)
  → Register / login (my-applications :3002 + API :4000)
  → Upload and parse CV
  → Apply to job (select CV, optional cover letter, screening answers)
  → Track status on dashboard

9.2 Staff: publish and hire

Login (backoffice :3001)
  → Create and publish job
  → Review applications (list or Kanban)
  → Move cards / PATCH status (with rules)
  → Schedule interview when needed
  → Progress to Offered → Hired (or Reject with reason)
  → Download CV / cover letter from detail page

9.3 Admin: maintain reference data

Login → Administration → CRUD on lookups (departments, skills, etc.)
